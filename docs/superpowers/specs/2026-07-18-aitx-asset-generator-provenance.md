# AITX Asset Generator + Provenance — Design Spec

Date: 2026-07-18
Status: for review

## Goal

Make every AITX brand asset **generated by a rule-encoded call** and **reproducible from a version-controlled recipe**. Two problems, one core:

1. **Rules in code.** The brand (palette, fonts, logo placement, layout, the reference atoms to pass) lives in code, not in a person's head or a one-off prompt. A generate call encodes the rules.
2. **Provenance.** Every generated asset ships a recipe: the exact prompt or template + inputs, the exact model, and every input reference (path + content hash). "How did you make that?" is answered by opening a file, and the asset can be regenerated.

This is the reusable core. The deterministic renderer, the image-model path, the portal's "how it was made" view, and the optional Agent-SDK experiment runner all build on it.

Grounding: Gary's encounter-site poster generator has the good bones (a typed, pure `buildPrompt(formData)` + a small typed asset library) but **no provenance** (nothing stored, nothing replayable) and **prose-described rules** (colors/fonts described in English to the model, not code-enforced). This spec keeps the bones and fills both gaps.

## The Recipe (the heart)

Every generated asset `foo.png` gets a sibling `foo.recipe.json`. It is the complete, version-controlled record needed to understand and regenerate the asset.

```ts
type Recipe = {
  recipeVersion: 1;
  asset: string;               // output path, relative to repo root
  createdAt: string;           // ISO 8601 (stamped by the caller, not inside pure code)
  generator: {
    kind: "image-model" | "deterministic";
    tool: string;              // e.g. "aitx-gen@0.1.0"
  };

  // --- image-model backend (illustrated assets: comic scenes, memes, founders) ---
  model?: string;              // "openai:gpt-image-2" | "gemini:nano-banana-pro"
  prompt?: string;             // the EXACT prompt string sent
  params?: { size?: string; quality?: string; background?: string };
  references?: {               // every input image passed to the model
    path: string;              // repo-relative
    sha256: string;            // content hash — proves which bytes were used
    role?: string;             // "logo" | "founders-two-shot" | "face:michael" ...
  }[];

  // --- deterministic backend (structured assets: flyers, social cards) ---
  template?: { id: string; version: string };   // e.g. { id: "flyer", version: "1.2.0" }
  inputs?: Record<string, unknown>;              // typed form data (event, date, venue, ...)
  fonts?: { family: string; weight: string; file: string; sha256: string }[];

  // --- shared ---
  ruleSet?: { id: string; version: string };     // the brand ruleset version applied
  notes?: string;
};
```

**Honesty about reproducibility (stated in the recipe's meaning, not hidden):**
- **Deterministic** assets are byte-stable: same `template.version` + `inputs` + `fonts` → the same pixels. Exact replay.
- **Image-model** assets are recipe-exact but not pixel-exact: gpt-image-2 exposes no seed, so the same prompt + model + verified reference bytes reproduce the *intent*, and output varies within the model's nondeterminism. The recipe still fully captures the input; that is the reproducibility that exists, and we do not pretend otherwise.

## The generate call

One module (`generator/`) exposes two backends behind a common result. Every call writes the asset AND its recipe; there is no way to generate without a recipe.

```ts
type GenerateResult = { assetPath: string; recipePath: string; recipe: Recipe };

// Image-model backend. Hashes each reference, sends prompt+refs to the model,
// writes asset + recipe.
generateImageAsset(spec: {
  outPath: string;
  model: "openai:gpt-image-2" | "gemini:nano-banana-pro";
  prompt: string;
  params?: { size?: string; quality?: string; background?: string };
  references: string[];                 // repo paths; the fn hashes them into the recipe
  roles?: Record<string, string>;       // optional path -> role labels
}): Promise<GenerateResult>;

// Deterministic backend. Renders a code template (next/og / satori) with typed
// inputs; brand rules (palette, fonts, logo placement) are IN the template code.
renderDeterministicAsset(spec: {
  outPath: string;
  templateId: string;                   // resolves to a registered template + version
  inputs: Record<string, unknown>;      // validated against the template's input schema
}): Promise<GenerateResult>;

// Re-run a recipe. Deterministic -> byte-stable; image-model -> re-sends the
// exact prompt+refs (verifying reference hashes still match; errors if an atom changed).
regenerate(recipePath: string): Promise<GenerateResult>;
```

**Prompt/template builders stay pure and testable** (the encounter pattern kept): a flyer template's `buildInputs()` / a meme's `buildPrompt(fields)` are pure functions with unit tests; the backend calls them, then generates.

## Two backends — when each

| Asset type | Backend | Why |
|---|---|---|
| Flyers, meetup/hackathon posters, social cards, event graphics, sticker text | **deterministic** (next/og / satori) | Text + layout + logo placement must be exact and on-brand. Code-enforced palette/fonts/logo. Byte-reproducible. Instant. No model cost. |
| Comic scenes, memes with illustrated subjects, founder/product renders | **image-model** (gpt-image-2, refs) | Needs illustration/likeness the atoms carry. Prompt + reference atoms. Recipe-recorded. |

A single finished piece can combine both later (illustrated background from the model, exact type/logo composited deterministically on top) — out of scope for v1, but the recipe format already supports recording a pipeline via `notes` + a future `steps[]`.

## Where recipes live + retrofit

- Recipes are committed next to their asset (`goldens/memes/uncle-sam-submit.png` → `.../uncle-sam-submit.recipe.json`). The diff of a recipe IS the change history of how an asset is made.
- **Retrofit:** the goldens and founder/venue/product atoms generated this session were made with known prompts + model (gpt-image-2) + references. Backfill recipes for them from that record where available; where a prompt wasn't preserved, write a partial recipe marked `notes: "prompt not preserved (pre-provenance)"` rather than a fake one.

## The Agent-SDK experiment layer (optional, on top)

The Agent SDK is NOT the generator and NOT welded to a UI. It is an optional natural-language runner *over* `generateImageAsset` / `renderDeterministicAsset` exposed as tools:

- "Generate 10 hackathon-flyer options, vary the headline and the sunburst color" → the agent calls the deterministic/image backend N times with varied inputs, each producing an asset + recipe.
- "Try the meme in navy instead of orange" → one more call, one more recipe.

Because each call writes a recipe, an agent-run batch of experiments is fully provenanced and replayable. The rules are still in code; the agent only explores the input space in language.

## Portal integration (answers "who is this for")

The portal shifts from a download shelf to **a generator + provenance browser**:
- Each asset card gains a "How it was made" view rendered from its recipe: the model, the prompt/template + inputs, and thumbnails of every reference atom.
- (Later) a Generate page that drives `renderDeterministicAsset` (instant) and `generateImageAsset` (~40s) from a form.
The consumer is the AITX organizer / chapter leader who needs an on-brand asset fast, plus partners who trust it because the recipe is right there.

## Non-goals (v1)

- No image-model seed control (gpt-image-2 has none; we record, not fake, reproducibility).
- No multi-step composite pipeline execution (format reserves room; not built).
- No hosted Agent-SDK service (local/optional only).
- No auth (portal Generate page auth is a later, separate concern).

## Open questions

- Recipe sidecar (`foo.recipe.json`) vs a single `recipes/` index — sidecar chosen for locality (diff travels with the asset); revisit if it clutters.
- Deterministic renderer: next/og (satori) in the portal app vs a standalone render script. Leaning next/og since the portal is already Next.js and it matches the encounter OG-card precedent.
- Which template to build first (hackathon flyer is the highest-value, most rule-heavy candidate).
